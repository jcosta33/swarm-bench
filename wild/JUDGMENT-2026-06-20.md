# Wild-set judgment — 2026-06-20

The wild tier runs **real** swarm-cli changes (the recent remediation commits) through `swarm review --json`
with their **real** task packets. Wild changes have no seeded ground truth, so each surfaced fact is judged
here by a human — **with the bias recorded: I (the framework author) wrote both the gate and the packets, so
this is self-measurement; the changes themselves are real, but the judge is not independent.**

The judgment classifies each surfaced fact as **artifact** (a property of the wild materialization, not the
gate's verdict on the change), **noise** (an effective false positive — a reviewer judging the *change* takes
no action on it, per [[GOOGLESA]]), or **true positive** (the gate surfaced a real property of the change a
reviewer would act on).

## Per-class judgment (across all 4 wild cases)

| Fact class | Count | Verdict | Why |
|---|---|---|---|
| `coverage/uncovered` | 10 | **artifact** | The wild cases ship no review packet, so `swarm review` (scan mode) reports every in-scope AC uncovered. A real review has a packet; excluded from the FP count. |
| `selfReport.inDiffNotClaimed` | 28 | **noise** | The real packets' `## Run summary` lists changed files in **prose** (`taskLocator, deriveBoard, …`); the claimed-files parser reads only backticked/path-like tokens, so it parses nothing claimable from prose (w3/w4) and flags the unclaimed files. (W1's one backticked path, `src/infra/yamlScalar.ts`, *was* parsed and correctly excluded — 8 of its 9 files surfaced, not 9.) The change is fine; the run-summary is just prose. |
| `selfReport.claimedNotInDiff` | 2 | **noise** | The only backticked token in two run-summaries was the **commit sha** (`` `0791385` ``) → mistaken for a claimed file path. A parser quirk, not a change issue. |
| `outsideScope` (co-located test files) | 13 | **noise** | The packets' `## Affected areas` list the `src/…` source paths but not the co-located `__tests__/` files; the gate flags the tests. Tests are expected with the change. |
| `outsideScope` `src/infra/yamlScalar.ts` (w1) | 1 | **true positive** | The W1 packet declared the helper at `Core/services/` but it landed at `infra/` (a real layering correction). The gate correctly surfaced a file changed **outside the declared areas**. |

## The number (the honest, less-biased result)

Excluding the no-packet coverage **artifacts**, the gate surfaced **44 diff-keyed facts** across the 4 real
changes. **1** was a true positive (the W1 scope divergence); **43** were effective false positives.

**Effective-FP ≈ 43/44 ≈ 98%** — versus the synthetic tier's **0%**.

The gap is the finding: the synthetic 0% was measured on **hand-built clean packets** (backticked file lists,
test dirs in `## Affected areas`, a covering review packet). The **real** packets used **prose run-summaries**
and didn't enumerate test dirs — and the gate's `selfReport` floods on exactly that. So the gate's real-world
precision is **contingent on packet discipline** (a machine-checkable run-summary + scoped affected-areas + a
review packet) and on the run-summary parser's robustness — neither of which the natural packets had.

## What this says (and does not)

- It does **not** say the gate is bad. The one true positive shows it catches real scope divergence; on a
  disciplined packet (the synthetic tier) it is silent. The check *facts* are sound.
- It **does** say the headline "0% effective-FP" is conditional, and a less-biased setting surfaces a real
  precision risk the synthetic corpus hid: **on a natural prose packet the gate is ~98% noise** and would be
  `--no-verify`'d. The fix is either packet-discipline enforcement (require backticked file lists / test dirs
  in scope) or a more robust run-summary parser — filed as a follow-up gate-precision finding.
- It is still self-measurement. A genuinely independent wild set (an outside team's PRs, judged by a
  non-author) is the next, less-biased data point beyond this one.
